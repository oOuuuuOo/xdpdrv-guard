# Architecture — xdpdrv-guard

This doc explains *why* the project is shaped the way it is. The README
tells you what to type; this tells you what to expect when you change
something.

---

## Two-tier defence

```
                ┌─ tier 1 ─────────────────────────────────────────┐
   wire ──────► │   XDP program at the network-card driver         │
                │   - hard, programmable, runs before sk_buff      │
                │   - cannot keep state cheaply (no map, today)    │
                └────────────────┬─────────────────────────────────┘
                                 │ XDP_PASS
                                 ▼
                ┌─ tier 2 ─────────────────────────────────────────┐
                │   nft chain `xdpdrv_guard_input`, prio -300      │
                │   - covers the early-boot window before XDP      │
                │   - covers any case the C parser doesn't handle  │
                │   - allows rate-limited drop on allowed ports    │
                └────────────────┬─────────────────────────────────┘
                                 ▼
                          your app / ssh / etc.
```

Why two tiers, not one:

- **The XDP layer can't always be there.** Between system boot and the
  moment systemd has run our `ExecStart`, the iface exists and accepts
  packets but no XDP program is loaded. nft (loaded earlier in the
  boot sequence) covers that gap.
- **The XDP layer drops decisions the verifier won't let you make
  cheaply.** The current implementation has no BPF map; rate-limit by
  source IP, GeoIP, conntrack state — all of those need a map and are
  delegated to nft.
- **nft alone leaks CPU.** Every dropped SYN climbs through the early
  netfilter chain. On a small VPS hitting a flood, that's measurable.
  XDP turns "drop in netfilter" into "drop in driver" — same packet,
  much less CPU.

So the two tiers are **complementary**, not redundant: nft for breadth
and early-boot, XDP for cheap volume.

## What's deliberately *out of scope*

- **Replacing your existing nftables.** The script writes ONLY to its
  own table `inet xdpdrv_guard`. Your `inet filter` and any business
  rules are untouched. If `down` runs, only `inet xdpdrv_guard` is
  removed.
- **Falling back to `xdpgeneric`.** The CPU cost of generic mode
  defeats the purpose. If `xdpdrv` doesn't load, the script fails
  loud — exit non-zero, leave nft tier as-is.
- **K8s / Cilium / Calico co-residency.** Those projects also want the
  XDP slot. Pick one. The script does not negotiate.
- **Big-iron rate scale.** The port table inside the eBPF program is
  static and walked O(N). For ≤ 100 port ranges it's fine. For 4-digit
  port counts, the verifier path explodes; AUDIT §2.1 will fix that
  with a 65536-bit BPF map.

## Data-plane: `xdp_syn_guard`

Generated by `generate_c_program()` from the active config and compiled
in place with `clang -target bpf`. The generator emits a deterministic
file; the same config produces byte-identical C.

### Packet decision tree

```
ethhdr
  ├── ETH_P_8021Q / ETH_P_8021AD ──► strip 1 VLAN tag, repeat (max 2 levels)
  ├── ETH_P_IP                    ──► IPv4 path
  ├── ETH_P_IPV6                  ──► IPv6 path
  └── other                       ──► XDP_PASS

IPv4 path
  ├── frag_off & 0x1FFF != 0      ──► XDP_PASS  (later fragment, no L4)
  ├── proto = TCP, SYN && !ACK    ──► dport ∉ ALLOWED_TCP_PORTS ? DROP : PASS
  ├── proto = UDP                 ──► dport ∉ ALLOWED_UDP_PORTS ? DROP : PASS
  └── other                       ──► XDP_PASS

IPv6 path
  ├── walk ext-header chain (HBH/RTH/DESTOPT, max depth 6)
  │     - FRAGMENT  ──► XDP_PASS
  │     - AH/ESP/MH ──► XDP_PASS
  ├── inner proto = TCP, SYN && !ACK ──► dport ∉ ALLOWED_TCP_PORTS ? DROP : PASS
  ├── inner proto = UDP              ──► dport ∉ ALLOWED_UDP_PORTS ? DROP : PASS
  └── other                          ──► XDP_PASS
```

### Why the bounds matter for the verifier

- **VLAN unwrap is `#pragma unroll` over 2 iterations.** Two fixed
  steps cover single-tagged 802.1Q and Q-in-Q / 802.1ad. A third level
  would require either another fixed iteration or a real loop, which
  the verifier handles less reliably across kernel versions.
- **IPv6 ext-header chain is `#pragma unroll` over 6 iterations.** Six
  is more than any sane real packet, and unroll keeps the verifier
  from synthesising loop bounds.
- **Allow-list lookup is linear.** For tens of ranges that's fine; for
  thousands it's not. The next architecture step (AUDIT §2.1) replaces
  the linear scan with a 65536-entry BPF map and lifts the rebuild
  requirement entirely.

### Why we re-attach on every config change (for now)

`generate_c_program()` rewrites the source, `compile_program()` rebuilds
the .o, then `cmd_install()` runs `detach_xdp_all_modes()` followed by
`attach_xdpdrv()`. Between detach and attach there is a sub-second
window with no XDP guard. The nft tier still drops in that window, but
the CPU floor is higher.

This is the **largest known limitation** of the current design and is
the headliner of AUDIT §1.5 / §2.1: switch to a BPF map-driven port
table and a single long-lived program, so a config change is a `bpf_map_update_elem` call, not a recompile + re-attach.

## Control plane: the bash script

The script does six independent things:

| Module | Functions / sections | Notes |
|---|---|---|
| Config loader | `parse_conf_file`, `verify_conf_perms`, `CONF_ALLOWED_KEYS` | Whitelist parser; refuses non-root / writable files; refuses metachars in values |
| Port arithmetic | `is_valid_port_token`, `expand_ports_csv_to_lines`, `compress_ports_lines_to_csv`, `csv_add_token_compact`, `csv_remove_token_compact`, `port_in_csv` | Pure bash; covered by bats tests in `tests/test_helpers.bats` |
| Data-plane builder | `generate_c_program`, `compile_program`, `attach_xdpdrv`, `detach_xdp_all_modes` | sha256-cached recompile; arch-aware include path |
| nft tier | `cmd_fw_install`, `cmd_fw_remove`, `cmd_fw_status`, `nft_table_exists` | Owns ONLY `inet xdpdrv_guard`; never touches user chains |
| fw-sync | `cmd_fw_sync_*`, `firewall_fingerprint`, `discover_firewall_allowed_tcp_ports` | Timer-driven; computes runtime view = baseline ∪ live host fw; never writes back |
| Reporting | `cmd_status`, `cmd_value_report`, `cmd_surface_audit`, `cmd_login_report` | Pure read; uses `nft -j` + jq when available |

Each module's contract is small enough that the long-term refactor
(AUDIT §2.6 split into `lib/`) is mechanical, not architectural.

## fw-sync: the two-layer model

```
   /etc/xdpdrv-guard.conf  (you own this; one source of truth)
                │  baseline = ALLOWED_TCP_PORTS
                ▼
   ┌─────────────────────────┐
   │  fw-sync timer (30 s)   │
   │  + host's nft/ipt/ufw   │ ────► live = ports accepting TCP today
   │    runtime snapshot     │
   └─────────────┬───────────┘
                 ▼
   runtime view = baseline ∪ live   (recomputed each tick — never stored)
                 │
                 ▼
   reload XDP + nft tier with runtime view
```

The pre-0.1.1 design merged `live` *into* the conf and persisted it.
That made the conf a monotonically-growing log of every port that had
ever been opened anywhere on the host, with no way to shrink it short
of editing by hand. The 0.1.1 design throws `live` away each tick and
recomputes from scratch, so removing a port from `ufw` makes it
disappear from the runtime view on the next tick — the way an operator
expects.

## Concurrency model

**One process at a time** for any mutating command (`cmd_install`,
`cmd_uninstall`, `cmd_fw_install`, `cmd_fw_sync_now`).
Enforced by `flock` on `/var/lock/xdpdrv-guard.lock` with a 30 s
timeout. Reentrant within a single process so that `cmd_install`
calling `cmd_fw_install` doesn't deadlock against itself.

Read-only commands (`cmd_status`, `cmd_doctor`, `cmd_value_report`,
`cmd_surface_audit`, `cmd_login_report`, `cmd_fw_status`,
`cmd_fw_sync_status`) deliberately don't take the lock. They tolerate
stale reads — at worst they print a snapshot from the brief moment
*during* an install transition.

Why not just rely on systemd's `Type=oneshot` to serialise? Because
the timer-driven `xdpdrv-guard-fw-sync.service` runs from a different
unit; without the file-system lock, a manual `up` and a timer-driven
sync could race on `$CONF_FILE` and the OBJ build dir.

## Failure model and rollback

`cmd_install` is the most consequential operation. As of 0.1.1:

1. **Generate** the C source from current config. (Pure local file write.)
2. **Compile** with clang. Failure here ends the install — neither nft
   nor XDP have been touched.
3. **Snapshot** the existing `inet xdpdrv_guard` table. Empty if first
   install.
4. **Apply** the new nft tier (`cmd_fw_install`).
5. **Detach + Attach** XDP. If attach fails:
   - delete the freshly-installed `xdpdrv_guard` table
   - replay the snapshot via `nft -f -`
   - return non-zero

So:

| Failure point | nft state | XDP state |
|---|---|---|
| Compile  | unchanged | unchanged |
| Apply    | new tier left in place (the fault was logged but apply itself succeeded) | unchanged |
| Attach   | rolled back to snapshot | detached |

The remaining gap: if `cmd_fw_install` itself fails mid-write
(e.g. `nft -f -` fatals on syntax error from a bad port token), the
table is in whatever state nft left it. We rely on the parser
upstream — `parse_ports_to_nft_set` validates each token before
assembling the rule — to make this case unreachable.

## Security boundary

The script runs as root. The boundary it defends is **not "what
someone with shell can do"** (that's already root-equivalent), but:

- **Tampering with the conf file** by lower-privileged users. Defended
  by `verify_conf_perms` (reject non-root or group/world-writable
  files) and `parse_conf_file` (whitelist, no `source`).
- **Privilege escalation via systemd unit attack surface**. Defended
  by `NoNewPrivileges`, `ProtectSystem=strict`, `ProtectHome=read-only`,
  `ProtectKernel*`, `RestrictAddressFamilies`, and a tight
  `CapabilityBoundingSet` covering only what BPF + netlink need.
- **Race-window state corruption**. Defended by the file-system lock.
- **Half-applied rollouts**. Defended by the snapshot+rollback path.

Things the script does NOT defend against and will not pretend to:

- Anything an existing root user does on the box.
- Kernel-level escapes via the BPF verifier itself (those are CVEs in
  the kernel, not in this project).
- Out-of-band traffic-class manipulation (tc/qdisc set by other tools).

## Where to look when changing things

| Want to change… | Read these first |
|---|---|
| What ports get dropped | `parse_l4_guard()` in the heredoc inside `generate_c_program()` |
| How the conf is parsed | `parse_conf_file` + `CONF_ALLOWED_KEYS` |
| The nft tier rules | `cmd_fw_install` |
| The fw-sync logic | `firewall_fingerprint`, `discover_firewall_allowed_tcp_ports`, `cmd_fw_sync_now` |
| Concurrency | `acquire_install_lock` + every `cmd_*` that calls it |
| systemd hardening | `write_systemd_unit` + `write_fw_sync_systemd_unit` |
| The MOTD path | `cmd_motd_install` heredoc + the `/run/xdpdrv-guard/motd.txt` cache |
| Tests | `tests/test_helpers.bats` |
| CI | `.github/workflows/ci.yml` |

## Roadmap pointers

This document describes the system as it is. For where it's going:

- 5–6 week sprint plan: AUDIT §5
- 6-month / 2-year horizons: AUDIT §7

The single biggest pending change is BPF-map-driven port tables (AUDIT
§1.5 / §2.1). That alone closes the re-attach window, removes the
clang dependency from the runtime path, and lifts the port-count
ceiling — three of the limitations called out in this document.
